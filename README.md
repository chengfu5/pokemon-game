# Pokemon AI Battle Simulator

A full-stack Pokemon battle simulator featuring a 6v6 tactical combat system, real-time machine learning win-probability prediction, and multiple front-end interfaces. Built with FastAPI, Pygame, and scikit-learn/XGBoost.

## Overview

This project is a Pokemon battle simulator that supports **Gen 1–4 Pokemon** (493 total) with authentic stat calculations, a full type-effectiveness chart (17 types), status conditions, abilities, and stat modifier stages. A trained ML model provides real-time **win-probability predictions** displayed during battle, giving players a data-driven view of their strategic position.

Three interfaces are available:
- **Web UI** — A browser-based battle screen served by FastAPI (`main.py`)
- **Desktop GUI** — A native Pygame window with sprite rendering (`gui_main.py`)
- **CLI** — A terminal-based text interface (`pokemon_game.py`)

## Technical Architecture

### Core Engine (`core/models.py`)

The battle engine is built around three primary classes:

#### `Move`
Represents a single attack or status move with the following attributes:
| Attribute | Type | Description |
|-----------|------|-------------|
| `name` | `str` | Display name (e.g., "Flamethrower") |
| `type` | `str` | One of 17 elemental types |
| `power` | `int` | Base power (0 for status moves) |
| `category` | `str` | `"Physical"`, `"Special"`, or `"Status"` |
| `accuracy` | `int` | Hit chance percentage (default 100) |
| `stat_change` | `dict` | Optional stat modifier (e.g., `{"stat": "attack", "stages": 2}`) |
| `status_effect` | `str` | Optional status infliction (`"BRN"`, `"PAR"`, `"PSN"`, `"SLP"`, `"Confusion"`) |

#### `Pokemon`
Each Pokemon is initialized from JSON data and computes Level 100 stats:

**Stat Formulas (Level 100, Perfect IVs, Balanced EVs):**
- **HP:** `(2 × Base + 31 + 85) + 110`
- **Other stats:** `(2 × Base + 31 + 85) + 5`

**Move Assignment:** On construction, each Pokemon receives 4 randomly selected moves from the master move database, filtered to only include STAB-eligible (same type) or Normal-type moves. This ensures every Pokemon has thematically appropriate coverage.

**Battle State:**
- `stat_stages` — Dictionary tracking stages (-6 to +6) for Attack, Defense, Sp.Atk, Sp.Def, and Speed
- `status` — Primary status condition (`"BRN"`, `"PAR"`, `"PSN"`, `"SLP"`, or `None`)
- `is_confused` — Volatile confusion flag (cleared on switch)

#### `Team`
Manages a roster of 6 Pokemon with switching logic, active index tracking, total/max HP aggregation, and alive count for ML feature extraction.

### Type Effectiveness System (`core/constants.py`)

A complete **Gen 4 type chart** covering all 17 types with three interaction layers:
- **Super effective** (2×) — e.g., Fire → Grass
- **Not very effective** (0.5×) — e.g., Fire → Water
- **Immunity** (0×) — e.g., Normal → Ghost

Dual-typed defenders have multipliers stacked multiplicatively (e.g., Ice move vs. Grass/Flying = 4×).

### Stat Stage Modifiers

Stat stages follow the official formula:
- Positive stages: `(2 + stage) / 2`
- Negative stages: `2 / (2 - stage)`
- Stages are clamped to the range [-6, +6]

### Damage Calculation (`calculate_damage`)

The damage formula is derived from the official Pokemon damage mechanics:

```
Base Damage = ((42 × Power × (A / D)) / 50) + 2
Final Damage = Base Damage × STAB × Type Multiplier × Ability Bonus × Burn Penalty
```

Where:
- **STAB** = 1.5× if the move type matches the attacker's type(s)
- **Ability Bonus** = 1.5× for Blaze/Torrent/Overgrow when HP ≤ 33%
- **Burn Penalty** = 0.5× on Physical moves when the attacker is burned
- **Levitate** = Ground-type moves automatically miss against Pokemon with Levitate

### Abilities

Currently implemented abilities:
| Ability | Effect |
|---------|--------|
| **Intimidate** | Lowers opponent's Attack by 1 stage on switch-in |
| **Levitate** | Grants immunity to Ground-type moves |
| **Blaze** | 1.5× Fire damage when HP ≤ 33% |
| **Torrent** | 1.5× Water damage when HP ≤ 33% |
| **Overgrow** | 1.5× Grass damage when HP ≤ 33% |

### Status Conditions

| Status | In-Battle Effect | End-of-Turn Effect |
|--------|------------------|--------------------|
| **BRN** (Burn) | Physical attack halved | Loses 1/8 max HP |
| **PAR** (Paralysis) | Speed quartered; 25% chance to skip turn | — |
| **PSN** (Poison) | — | Loses 1/8 max HP |
| **SLP** (Sleep) | Cannot act; 33% wake chance per turn | — |
| **Confusion** | 50% snap-out chance; 50% self-hit (40 BP typeless) | — |

Type immunities apply: Fire-types cannot be burned, Electric-types cannot be paralyzed.

### Web Server (`main.py`)

A **FastAPI** application serving both the static web UI and a RESTful game API. Battle state is stored in-memory in the `ACTIVE_BATTLES` dictionary keyed by UUID.

**Turn Resolution Order:**
1. Actions are sorted by priority (switches always go first) then by Speed stat
2. Status checks execute in order: Sleep → Confusion → Paralysis
3. Move execution (accuracy, damage, secondary effects)
4. Faint checks after each action
5. End-of-turn residual damage (Burn, Poison)

### Web UI (`static/`)

A single-page application built with **vanilla HTML/CSS/JavaScript**:
- **Lobby Screen** — Displays all 493 Pokemon in a scrollable grid for team drafting (pick exactly 6)
- **Opponent Selection** — Choose from legacy teams (iconic trainers) or a random team
- **Battle Screen** — An 800×600 pixel canvas replicating the Nintendo DS battle layout with:
  - Front/back sprite rendering
  - Animated HP bars with color transitions (green → yellow → red)
  - Real-time ML win probability tracker
  - Step-by-step event playback with 1.5s delays between snapshots

### Desktop GUI (`gui_main.py`)

A **Pygame-CE** implementation of the same battle system with:
- Cached sprite loading and scaling (3× for player, 2.5× for opponent)
- Frame-based message queue system (90 frames = 1.5s per message at 60 FPS)
- State machine managing menu transitions: `MAIN_MENU → FIGHT_MENU → PROCESS_TURN → MESSAGE_STATE`
- Forced-switch handling when a Pokemon faints mid-turn

### Legacy Teams (`core/constants.py`)

Pre-built teams inspired by iconic trainers from the anime and games:
- **Cynthia** (Sinnoh Champion) — Spiritomb, Roserade, Gastrodon, Lucario, Milotic, Garchomp
- **Ash Ketchum** (Kanto Classic) — Pikachu, Charizard, Squirtle, Bulbasaur, Pidgeot, Snorlax
- **Ash Ketchum** (Sinnoh) — Pikachu, Infernape, Torterra, Staraptor, Buizel, Gliscor
- **Gary Oak** — Pidgeot, Alakazam, Rhydon, Exeggutor, Arcanine, Blastoise
- **Red** (Mt. Silver) — Pikachu, Espeon, Snorlax, Venusaur, Charizard, Blastoise
- **Lance** (Johto Champion) — Gyarados, Flygon, Aerodactyl, Salamence, Charizard, Dragonite

## Gameplay Mechanics

### Battle Flow

1. **Team Drafting** — Select 6 Pokemon from the full Gen 1–4 Pokedex (493 Pokemon)
2. **Opponent Selection** — Pick a legacy trainer team or face a random team
3. **Battle Phase** — 6v6 single battle with turn-based combat
4. **Victory Condition** — First player to knock out all 6 opposing Pokemon wins

### Turn Structure

Each turn follows this sequence:
1. **Decision Phase** — Player chooses to Fight (select a move) or Switch (select a bench Pokemon)
2. **AI Decision** — The opponent selects a move at random from its active Pokemon's move pool
3. **Priority Resolution** — Switches always execute before attacks; ties broken by Speed stat
4. **Action Execution** — Each action resolves sequentially with status checks, damage, and faint handling
5. **End-of-Turn** — Residual damage from Burn and Poison is applied

### Move System

Each Pokemon is assigned **4 moves** at battle start, chosen randomly from the 467-move master database with the following constraints:
- Moves must match one of the Pokemon's types (STAB) or be Normal-type
- Damaging moves must have >0 base power
- Status moves (power = 0) are included if categorized as "Status"

Moves are split into three categories:
- **Physical** — Uses Attack vs. Defense
- **Special** — Uses Sp.Atk vs. Sp.Def
- **Status** — Applies stat changes or status conditions (no direct damage)

### AI Behavior

- **In-game AI** — Currently selects moves randomly from the active Pokemon's move pool
- **Simulation AI** (`simulate_battles.py`) — Uses a greedy strategy picking the highest-damage move, with a 5% random switch chance to simulate human unpredictability

## Machine Learning Pipeline

### Overview

The ML system provides real-time **win-probability predictions** during battle, displayed as a percentage tracker in both the web and Pygame interfaces. The pipeline follows a three-stage process: data generation → model training → live inference.

### Stage 1: Data Generation (`simulate_battles.py`)

Simulates **1,000 bot-vs-bot matches** using the greedy AI to generate labeled training data:

- Each turn produces a feature snapshot capturing the board state
- All snapshots in a match are labeled with the final outcome (Player 1 won = 1, lost = 0)
- A 200-turn limit prevents infinite loops
- Output: `data/battle_dataset.csv`

**Feature Vector (per turn snapshot):**
| Feature | Description |
|---------|-------------|
| `p1_hp_pct` | Player 1's total remaining HP as a fraction of max HP |
| `p1_alive` | Number of non-fainted Pokemon on Player 1's team |
| `p2_hp_pct` | Player 2's total remaining HP as a fraction of max HP |
| `p2_alive` | Number of non-fainted Pokemon on Player 2's team |
| `p1_won` | Target label — 1 if Player 1 won the match, 0 otherwise |

```bash
python simulate_battles.py   # Generates ~50,000+ turn snapshots
```

### Stage 2: Model Training (`ai/train_models.py`)

A **model bake-off** using `RandomizedSearchCV` for hyperparameter tuning across three algorithms:

| Model | Hyperparameter Search Space |
|-------|----------------------------|
| **Logistic Regression** | C ∈ {0.01, 0.1, 1, 10, 100} |
| **Random Forest** | n_estimators ∈ {50, 100, 200}, max_depth ∈ {None, 5, 10, 20}, min_samples_split ∈ {2, 5, 10} |
| **XGBoost** | n_estimators ∈ {50, 100, 200}, learning_rate ∈ {0.01, 0.1, 0.2}, max_depth ∈ {3, 5, 7}, subsample ∈ {0.8, 1.0} |

- **Cross-validation:** 3-fold CV with up to 10 random hyperparameter combinations per model
- **Evaluation:** Accuracy + ROC-AUC on an 80/20 train/test split
- **Output:** The best-performing model is serialized to `data/best_battle_model.pkl` via `joblib`

```bash
cd ai && python train_models.py
```

### Stage 3: Live Inference (`ai/ml_tracker.py`)

The `WinProbabilityModel` class loads the serialized model at server startup and provides predictions on each turn:

1. Extracts the 4-feature vector from the current game state
2. Calls `model.predict_proba()` to get P(player wins)
3. Returns the probability as a float [0, 1]

**Fallback Heuristic:** If no trained model is found, a sigmoid-based heuristic is used:
```
z = (hp_diff × 2.5) + (roster_diff × 1.5)
P(win) = 1 / (1 + e^(-z))
```

This ensures the ML tracker always displays a meaningful value, even without a trained model.

## API Reference

### `GET /`
Serves the main HTML battle page.

### `GET /api/pokedex`
Returns a list of all 493 Pokemon with `id`, `name`, and `types` for the drafting screen.

### `GET /api/legacy_teams`
Returns a list of available legacy team names (e.g., `"Cynthia (Sinnoh Champion)"`).

### `POST /api/start_battle`
Creates a new battle instance.

**Request Body:**
```json
{
  "player_team_names": ["Charizard", "Blastoise", "Venusaur", "Pikachu", "Snorlax", "Gengar"],
  "opponent_name": "Cynthia (Sinnoh Champion)"
}
```

**Response:**
```json
{
  "battle_id": "uuid-string"
}
```

### `POST /api/action`
Executes a turn and returns step-by-step event snapshots.

**Request Body:**
```json
{
  "battle_id": "uuid-string",
  "action_type": "attack",
  "target_idx": 0
}
```

**Action Types:**
| Type | Description | `target_idx` |
|------|-------------|--------------|
| `init` | Initialize battle state | 0 |
| `attack` | Use a move | Move index (0–3) |
| `switch` | Voluntarily switch Pokemon | Roster index (0–5) |
| `forced_switch` | Forced switch after a faint | Roster index (0–5) |

**Response:** An array of event snapshots, each containing:
- Board state (HP, names, statuses for both sides)
- ML win probability
- Available moves and roster info
- Game-over flag

## Future Enhancements

### Move PP (Power Points)
Currently, moves have unlimited uses. A future update would add the authentic PP system where each move has a limited number of uses per battle (e.g., Flamethrower has 15 PP, Hyper Beam has 5 PP). When a move's PP reaches 0, it becomes unusable. If all moves are depleted, the Pokemon resorts to **Struggle** — a typeless 50 BP recoil move that damages the user for 25% of their max HP. This adds a critical resource-management layer to longer battles.

### Smarter Move Selection
The current system randomly assigns 4 STAB/Normal-type moves to each Pokemon. Future improvements include:
- **Learnset-Accurate Movesets** — Only assign moves that the Pokemon can actually learn according to official game data (the `learnable_moves` field is already fetched and stored in `pokemon_data.json`)
- **Competitive Moveset Templates** — Curate optimized movesets for each Pokemon based on competitive tier lists (e.g., Smogon sets)
- **Coverage Balancing** — Ensure each Pokemon has a mix of STAB, coverage, and status moves for strategic depth rather than purely random selection
- **Move Priority** — Implement priority brackets (e.g., Quick Attack at +1, Protect at +4) for moves that bypass normal Speed ordering

### Smarter AI Opponent
The current AI selects moves randomly. Planned improvements:
- **Type-Aware AI** — Evaluate type matchups before selecting a move, preferring super-effective hits
- **Predictive Switching** — AI switches out when at a severe type disadvantage
- **ML-Driven Decisions** — Train a reinforcement learning agent that uses the win-probability model to choose optimal actions
- **Difficulty Tiers** — Offer Easy (random), Medium (greedy), and Hard (ML-driven) AI settings

### Expanded Pokemon Generations
The current dataset covers Gen 1–4 (Pokemon #1–493). Extending to later generations would add:
- **Gen 5 (Unova)** — Pokemon #494–649, introducing 156 new species
- **Gen 6 (Kalos)** — Pokemon #650–721, adding Fairy type (18th type) and Mega Evolutions
- **Gen 7 (Alola)** — Pokemon #722–809, adding regional forms and Z-Moves
- **Gen 8–9 (Galar/Paldea)** — Pokemon #810–1025, adding Dynamax and Terastallization

Each generation would require updating the type chart (Fairy type), fetching new sprites, and expanding the move database.

### Additional Battle Mechanics
- **Held Items** — Implement items like Choice Band (1.5× Attack, locked into one move), Leftovers (1/16 HP recovery per turn), and Life Orb (1.3× damage, 10% recoil)
- **Weather Effects** — Sun (1.5× Fire, 0.5× Water), Rain (1.5× Water, 0.5× Fire), Sandstorm (Rock/Ground/Steel Sp.Def boost + residual damage), Hail (residual damage to non-Ice types)
- **Entry Hazards** — Stealth Rock (type-based switch-in damage), Spikes (layered switch-in damage), Toxic Spikes (auto-poison on switch)
- **Critical Hits** — ~6.25% chance for 1.5× damage, ignoring negative Attack/positive Defense stages
- **Accuracy/Evasion** — Implement move accuracy checks (currently all moves hit) and evasion stat stages
- **Multi-Hit Moves** — Support moves like Double Slap (2–5 hits) and Bullet Seed
- **Recoil Moves** — Implement self-damage on moves like Brave Bird (33% recoil) and Double-Edge (33% recoil)
- **Nature System** — Each Pokemon gets a nature that boosts one stat by 10% and reduces another by 10%

### ML Model Improvements
- **Richer Feature Set** — Include type matchup scores, stat stage modifiers, status conditions, active Pokemon speed tier, and remaining move coverage in the feature vector
- **Turn-Level Predictions** — Predict the optimal action (attack/switch + target) rather than just win probability
- **Deep Learning** — Explore neural network architectures (LSTM/Transformer) to capture sequential turn dependencies
- **Online Learning** — Update the model incrementally from live player battles rather than only pre-simulated data
- **Opponent Modeling** — Track and predict opponent tendencies across multiple battles

### UI/UX Improvements
- **Battle Animations** — Add attack animations, damage shake effects, and faint transitions
- **Sound Effects** — Implement Pokemon cries, move sound effects, and background music
- **Move Tooltips** — Show detailed move info (PP, accuracy, secondary effects) on hover
- **Battle Log** — A scrollable text history of all events in the current battle
- **Team Builder** — A dedicated team-building interface with stat visualization, type coverage analysis, and moveset planning
- **Multiplayer** — WebSocket-based PvP battles between two human players
